# hello-rag.mjs 代码笔记

## 一、整体架构：RAG（检索增强生成）

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  文档入库  │ ──▶ │  向量检索  │ ──▶ │  LLM 生成 │
│ (Embedding)│     │ (Retrieve)│     │ (Generate)│
└──────────┘     └──────────┘     └──────────┘
```

---

## 二、使用的模型

| 用途 | 模型 | 来源 |
|------|------|------|
| 对话生成 | `qwen-max` | 阿里云 DashScope（OpenAI 兼容接口） |
| 文本嵌入 | `text-embedding-v3` | 同上（建议升级到 `text-embedding-v4`） |

> 代码通过 `@langchain/openai` 调用，但 `baseURL` 指向了 `dashscope.aliyuncs.com/compatible-mode/v1`，实际调用的是阿里云百炼平台。

## 三、核心流程拆解

### 1. 初始化模型实例

```js
// 对话模型
const model = new ChatOpenAI({ temperature: 0, model: 'qwen-max', ... });

// 嵌入模型
const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-v3', ... });
```

### 2. 准备知识库文档（7 篇小故事）

```js
const documents = [
  new Document({
    pageContent: '文档正文...',
    metadata: { chapter: 1, character: '光光', type: '角色介绍', mood: '活泼' }
  }),
  // ... 共 7 篇
];
```

`metadata` 用于后续过滤和展示，**不参与向量检索**。

### 3. 文档向量化 + 入向量库

```js
const vectorStore = await MemoryVectorStore.fromDocuments(documents, embeddings);
```

- 调用嵌入模型，把每篇文档转成高维向量
- `MemoryVectorStore`：向量存内存，重启即丢失，仅适合 demo

### 4. 创建检索器

```js
const retriever = vectorStore.asRetriever({ k: 3 });
```

`k: 3` 表示每次检索返回最相似的 3 篇文档。

### 5. 检索 + 生成答案

```js
// 用户问题
const question = "东东和光光是怎么成为朋友的？";

// 向量检索：把问题也转成向量，在库中找最相似的文档
const retrievedDocs = await retriever.invoke(question);

// 组装 prompt：检索到的文档作为上下文
const prompt = `基于以下故事片段回答问题...

故事片段: ${context}
问题: ${question}`;

// 交给 LLM 生成最终答案
const response = await model.invoke(prompt);
```

## 四、关键技术点

### Embedding（嵌入）的作用

| | 文本 | 向量表示 |
|------|------|------|
| 文档 A | "光光擅长踢足球" | `[0.12, -0.34, 0.78, ...]` |
| 问题 Q | "东东和光光怎么认识的" | `[0.11, -0.31, 0.75, ...]` |

> 语义越相似，向量之间的余弦距离越小 → 检索越精准。

### MemoryVectorStore vs 生产环境

| 方案 | 特点 |
|------|------|
| `MemoryVectorStore` | 内存存储，每次重启重建索引 |
| `Chroma` / `Pinecone` / `Milvus` | 持久化，支持海量数据 |

### 相似度评分

代码额外调用了 `similaritySearchWithScore` 获取评分：
- 返回的是**距离**（越小越相似）
- 用 `1 - score` 转换为相似度

---

## 五、改进建议

1. **嵌入模型升级**：`text-embedding-v3` → `text-embedding-v4`（v3 已不在百炼推荐列表）
2. **持久化向量库**：替换 `MemoryVectorStore` 为 `Chroma` 或 `FAISS`，避免每次重启重建索引
3. **批量缓存**：如果文档不会变，缓存 embedding 结果，避免重复调用 API
4. **元数据过滤**：检索时可按 `mood`、`type` 等 metadata 先过滤，提高精度

# loader-and-splitter.mjs / loader-and-splitter2.mjs 笔记

## 一、对比 hello-rag.mjs 的区别

| | hello-rag.mjs | loader-and-splitter 系列 |
|------|------|------|
| **文档来源** | 代码硬编码 7 篇故事 | **爬取网页**（掘金文章） |
| **文档处理** | 整篇直接入库 | **先切分成小块**再入库 |
| 适用场景 | 固定知识库 | 动态网页内容 |

## 二、新引入的概念

### 1. CheerioWebBaseLoader：网页加载器

```js
const cheerioLoader = new CheerioWebBaseLoader(
  "https://juejin.cn/post/7233327509919547452",
  { selector: '.main-area p' }  // 只提取正文区域的 <p> 标签
);
const documents = await cheerioLoader.load();
```

- 用 Cheerio（服务端 jQuery）解析 HTML
- `selector` 过滤噪音（导航栏、评论区等）

### 2. RecursiveCharacterTextSplitter：文本切割器

```js
const textSplitter = new RecursiveCharacterTextSplitter({
  chunkSize: 400,       // 每块最多 400 字符
  chunkOverlap: 50,     // 相邻块重叠 50 字符
  separators: ["。","！","？"],  // 优先在句号/感叹号处断开
});
```

| 参数 | 作用 | 太小 | 太大 |
|------|------|------|------|
| `chunkSize` | 每块大小 | 语义碎片化 | 检索不精准 |
| `chunkOverlap` | 重叠量 | 冗余多 | 关键信息可能被切断 |

> 切割策略：按 `separators` 优先级递归切割——先找 `。`，找不到再用下一种分隔符。

### 3. loader-and-splitter vs loader-and-splitter2

| | loader-and-splitter | loader-and-splitter2 |
|------|------|------|
| 功能 | 仅演示加载+切割 | 完整 RAG 流程 |
| 输出 | `console.log(splitDocuments)` | 检索结果 + AI 回答 |

## 三、完整网页 RAG 流程

```
网页 URL
  │  CheerioWebBaseLoader（爬取 + 解析 HTML）
  ▼
整篇文档
  │  RecursiveCharacterTextSplitter（按句子切分）
  ▼
N 个分块（chunks）
  │  OpenAIEmbeddings（每个分块 → 向量）
  ▼
MemoryVectorStore（向量库）
  │  similaritySearchWithScore（检索 top-k）
  ▼
拼接 prompt → LLM 生成答案
```

## 四、关键要点

1. **网页 RAG 必须切割**：一整篇网页太长，LLM 上下文塞不下且检索精度差
2. **`chunkOverlap` 防止信息断裂**：关键句可能刚好被切断，重叠保证上下文连续
3. **`selector` 决定数据质量**：不设选择器可能爬进导航、广告等垃圾内容
